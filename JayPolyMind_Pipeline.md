# Как работает пайплайн JayPolyMind — пошаговое объяснение

---

## Общая схема

```
Пользователь загружает файл + вопрос
           │
     ┌─────▼──────┐
     │   ШАГ 1    │  Граф знаний
     │  Онтология │  LLM читает документ → придумывает типы сущностей
     └─────┬──────┘
           │
     ┌─────▼──────┐
     │   ШАГ 2    │  Построение графа
     │  Neo4j     │  LLM извлекает сущности и связи из текста
     └─────┬──────┘
           │
     ┌─────▼──────┐
     │   ШАГ 3    │  Подготовка симуляции
     │  Агенты    │  Каждый узел графа → персонаж с психологическим профилем
     └─────┬──────┘
           │
     ┌─────▼──────┐
     │   ШАГ 4    │  Симуляция (OASIS)
     │  Twitter+  │  Агенты 72 часа пишут посты, лайкают, спорят
     │  Reddit    │
     └─────┬──────┘
           │
     ┌─────▼──────┐
     │   ШАГ 5    │  Отчёт + Диалог
     │  ReACT     │  LLM читает события и пишет аналитику
     └────────────┘
```

## ШАГ 1 — Граф знаний: что такое "онтология" и зачем она нужна

### Что делает пользователь
Загружает документ (PDF/MD/TXT) и вводит вопрос, например:
> «Как сотрудники и профсоюзы отреагируют на переход к 4-дневной неделе?»

### Что происходит внутри
Файл: `backend/app/services/ontology_generator.py`

**1.** Текст из файла извлекается и очищается (`FileParser` + `TextProcessor`)

**2.** LLM получает **один большой промпт**:
```
Ты — эксперт по онтологиям. Прочитай документ и придумай:
- 8 специфических типов сущностей (кто/что есть в этом тексте)
- + 2 запасных: "Person" и "Organization" (для всего, что не подходит)
- 6-10 типов связей между сущностями

Документ: [весь текст файла]
Вопрос пользователя: [вопрос]
```

**3.** LLM отвечает JSON-ом:
```json
{
  "entity_types": [
    { "name": "Employee",     "description": "Работник предприятия" },
    { "name": "TradeUnion",   "description": "Профсоюзная организация" },
    { "name": "Manager",      "description": "Руководящий персонал" },
    { "name": "Person",       "description": "Запасной тип для любого человека" },
    { "name": "Organization", "description": "Запасной тип для любой организации" }
  ],
  "edge_types": [
    { "name": "MEMBER_OF",   "description": "Является членом" },
    { "name": "MANAGES",     "description": "Управляет" },
    { "name": "OPPOSES",     "description": "Противостоит" }
  ]
}
```

### Зачем это нужно
Онтология — это **словарь понятий** для конкретного документа. Без неё система не знала бы, что именно искать в тексте. С ней — знает, что в этом документе есть "Employee", "TradeUnion", и что между ними бывает связь "OPPOSES".

## ШАГ 2 — Построение графа: превращение текста в базу данных

### Файл: `backend/app/services/graph_builder.py`

**1.** Текст режется на **куски по ~500 символов** с перекрытием 50 символов (чтобы не потерять смысл на стыках)

**2.** Каждый кусок отправляется в LLM с промптом:
```
Вот онтология (типы сущностей и связей).
Вот фрагмент текста.
Извлеки из него все сущности и связи строго по онтологии.
Верни JSON.

Фрагмент: "Иван Петров — председатель профсоюза НПЗ.
Он выступает против сокращения смен..."
```

**3.** LLM возвращает:
```json
{
  "nodes": [
    { "name": "Иван Петров",  "type": "Person",     "summary": "Председатель профсоюза НПЗ" },
    { "name": "Профсоюз НПЗ", "type": "TradeUnion", "summary": "Профсоюзная организация НПЗ" }
  ],
  "edges": [
    {
      "from": "Иван Петров",
      "to":   "Профсоюз НПЗ",
      "type": "MEMBER_OF",
      "fact": "Иван Петров — председатель профсоюза НПЗ"
    }
  ]
}
```

**4.** Узлы и рёбра сохраняются в **Neo4j** (графовая база данных)

### Как выглядит граф в Neo4j
```
(Иван Петров: Person) ──[MEMBER_OF]──► (Профсоюз НПЗ: TradeUnion)
(Профсоюз НПЗ: TradeUnion) ──[OPPOSES]──► (Директор завода: Manager)
(Мария Сидорова: Employee) ──[MEMBER_OF]──► (Профсоюз НПЗ: TradeUnion)
```

**Итог:** документ превращён в структурированную сеть людей, организаций и их взаимоотношений.

## ШАГ 3 — Подготовка симуляции: узлы графа становятся персонажами

Этот шаг самый насыщенный — он делает три вещи параллельно.

### 3А — Фильтрация сущностей
Файл: `backend/app/services/entity_reader.py`

Из Neo4j читаются все узлы. **Отбрасываются** узлы с меткой только `["Entity"]` (технический мусор). **Берутся** узлы с меткой `["Entity", "Employee"]`, `["Entity", "TradeUnion"]` и т.д.

Для каждого узла система собирает **контекст**: какие рёбра от него исходят, с кем он связан, что о нём написано в соседних узлах.

### 3Б — Генерация профилей агентов (самое важное)
Файл: `backend/app/services/oasis_profile_generator.py`

**Для каждой сущности** LLM создаёт полноценный психологический профиль:

**Входные данные для LLM:**
```
Сущность: Иван Петров
Тип: Person
Описание: Председатель профсоюза НПЗ, выступает против сокращения смен
Связи: MEMBER_OF → Профсоюз НПЗ, OPPOSES → Директор завода
Контекст из графа: [факты из связанных узлов]

Сгенерируй детальный психологический профиль этого персонажа
для симуляции социальных сетей. Пиши на русском языке.
```

**Что генерирует LLM:**
```json
{
  "persona": "Иван Петров — опытный профсоюзный деятель 52 лет.
              Прагматик с жёстким переговорным стилем.
              Не доверяет управленческим инициативам без консультаций с коллективом.
              В социальных сетях пишет взвешенно, но настойчиво...",
  "bio":       "Председатель профсоюза НПЗ, 20 лет стажа",
  "age":       52,
  "gender":    "male",
  "mbti":      "ISTJ",
  "country":   "RU",
  "profession": "Профсоюзный деятель",
  "interested_topics": ["трудовые права", "производство", "коллективные договоры"]
}
```

**Профили сохраняются в двух форматах:**
- `reddit_profiles.json` — для симуляции Reddit
- `twitter_profiles.csv` — для симуляции Twitter

### 3В — Конфигурация симуляции
Файл: `backend/app/services/simulation_config_generator.py`

LLM генерирует **правила поведения** для всей симуляции. Делается **поэтапно** (4 отдельных запроса к LLM), чтобы не превысить лимит токенов:

**Запрос 1 — Временная конфигурация:**
```json
{
  "total_simulation_hours": 72,
  "peak_hours": [20, 21, 22, 23],
  "peak_activity_multiplier": 1.5,
  "off_peak_hours": [0, 1, 2, 3, 4, 5],
  "off_peak_activity_multiplier": 0.05
}
```
→ Агенты активны вечером (московское время), почти не активны ночью.

**Запрос 2 — События:**
```json
{
  "initial_posts": [
    { "agent_id": 1, "content": "Начальный пост, запускающий дискуссию", "timestamp": 0 }
  ],
  "hot_topics": ["4-дневная неделя", "трудовые права", "производительность"],
  "narrative_direction": "Исследовать реакцию на изменение рабочего расписания"
}
```

**Запрос 3 — Поведение каждого агента (батчами по 15):**
```json
{
  "agent_id":       1,
  "activity_level": 0.8,
  "posts_per_hour": 0.5,
  "stance":         "opposing",
  "sentiment_bias": -0.4,
  "active_hours":   [8, 9, 10, 12, 18, 19, 20, 21]
}
```
→ Иван Петров активен, настроен оппозиционно, негативный сентимент.

**Запрос 4 — Параметры платформ:**
```json
{
  "platform":          "twitter",
  "recency_weight":    0.4,
  "popularity_weight": 0.3,
  "viral_threshold":   10
}
```

## ШАГ 4 — Симуляция: агенты "живут" 72 часа

### Как работает OASIS
Файл: `backend/scripts/run_parallel_simulation.py`

OASIS — это **движок симуляции социальных сетей**. Он запускается как отдельный процесс.

**Механика каждого раунда (1 раунд = 1 час симуляции):**

1. Система определяет, **какие агенты активны** в этот час (по `active_hours` из конфига)
2. Для каждого активного агента:
   - Агент **видит** ленту (посты от других агентов, отсортированные по весам recency/popularity/relevance)
   - LLM решает, что агент **сделает**: напишет пост / прокомментирует / лайкнет / проигнорирует
   - Решение принимается на основе **персоны** агента и его **stance** (позиция)
3. Все действия записываются в `actions.jsonl`

**Пример записи в actions.jsonl:**
```jsonl
{"round":1,"agent_id":1,"agent_name":"ivan_petrov_312","action_type":"CREATE_POST","content":"Четырёхдневная неделя — это красивые слова. Кто будет выполнять план?","timestamp":"2025-01-01T20:00:00"}
{"round":1,"agent_id":5,"agent_name":"maria_sidorova_891","action_type":"LIKE_POST","post_id":"post_001","timestamp":"2025-01-01T20:05:00"}
{"round":2,"agent_id":3,"agent_name":"director_kozlov_447","action_type":"CREATE_POST","content":"Производительность возрастёт. Международный опыт это подтверждает.","timestamp":"2025-01-01T21:00:00"}
```

**Мониторинг в реальном времени:**
`SimulationRunner` читает `actions.jsonl` на лету и отправляет обновления на фронтенд через **WebSocket**: текущий раунд, количество действий, активные агенты.

## ШАГ 5А — Генерация отчёта: паттерн ReACT

### Что такое ReACT
Файл: `backend/app/services/report_agent.py`

ReACT = **Re**ason + **Act**. LLM не просто "пишет отчёт" — он рассуждает и сам решает, какие данные ему нужны, запрашивает их из графа, анализирует, потом пишет.

### Цикл для каждого раздела отчёта:

```
┌─ THOUGHT ───────────────────────────────────────────┐
│ "Для раздела 'Ключевые участники' мне нужно знать   │
│  кто самый влиятельный, какие у них позиции."        │
└─────────────────────┬───────────────────────────────┘
                      │
┌─ TOOL CALL ──────────▼──────────────────────────────┐
│ search_graph("ключевые стейкхолдеры, влияние")       │
│ → возвращает: факты из Neo4j + действия из OASIS     │
└─────────────────────┬───────────────────────────────┘
                      │
┌─ OBSERVATION ────────▼──────────────────────────────┐
│ Получено 12 фактов о 5 агентах. Иван Петров          │
│ написал 23 поста, получил 89 лайков. Директор        │
│ Козлов — 8 постов, 12 лайков.                        │
└─────────────────────┬───────────────────────────────┘
                      │
┌─ REFLECTION ─────────▼──────────────────────────────┐
│ Данных достаточно? Да → писать раздел.               │
│ Данных мало? → ещё один TOOL CALL (до 5 итераций)   │
└─────────────────────┬───────────────────────────────┘
                      │
               НАПИСАТЬ РАЗДЕЛ
```

**Инструменты, которые LLM вызывает сам:**

| Инструмент | Что делает |
|-----------|-----------|
| `search_graph(query)` | Гибридный поиск: семантический (векторный) + BM25 по Neo4j |
| `get_entity_details(uuid)` | Возвращает полную карточку сущности со всеми связями |
| `get_statistics()` | Сколько узлов, рёбер, типов сущностей в графе |

**Структура отчёта (5–8 разделов):**
1. Предыстория и контекст
2. Ключевые участники и стейкхолдеры
3. Развитие нарратива и таймлайн событий
4. Паттерны взаимодействий и влияние
5. Распространение информации
6. Динамика настроений
7. Выводы и рекомендации

## ШАГ 5Б — Интерактивный диалог

После отчёта пользователь может задавать вопросы в чате. Каждый вопрос:
1. Вместе с историей чата и контекстом симуляции уходит в LLM
2. LLM **автономно** решает: нужен ли поиск по графу, или можно ответить из уже известного
3. Если нужен поиск — вызывает `search_graph()` и включает факты в ответ

## Ключевые принципы всей системы

| Принцип | Как реализован |
|---------|---------------|
| **Граф = память** | Neo4j хранит всё, что знает система о предметной области |
| **LLM = интерпретатор** | Не хранит данные, только интерпретирует и генерирует |
| **Профиль = персонаж** | Каждый узел графа → полноценный агент со своей психологией |
| **Конфиг = правила мира** | LLM задаёт "физику" симуляции под конкретный кейс |
| **ReACT = думай перед тем как писать** | LLM сначала собирает факты, потом синтезирует |
| **actions.jsonl = летопись** | Все действия агентов записаны, ничего не теряется |
